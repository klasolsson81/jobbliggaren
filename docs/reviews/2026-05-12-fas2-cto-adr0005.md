## Runda 1 — Huvudval (A/B/C)

### Beslut

**Alternativ C — Invite-only public beta med hård cap.**

Nyckel-mekanik:
- JobAd-listning/sökning auth-gated från Fas 2-start (ej anonym publik katalog)
- `registrations_open`-flaggan i `appsettings.json` + Secrets Manager override (BUILD.md §13.2)
- Två nya rate-limit-policies utöver befintliga tre (uppdaterade i Runda 2)
- AWS Budget Actions $50/mån auto-disable Bedrock-IAM + stop ECS
- Runbook minimum-set: budget-alert-respons, manuell flag-toggle, IAM-policy-återställning, ECS-restart
- AI-kostnadstak (TD-26) förblir Fas 4-bundet — ingen AI-yta i Fas 2

### Motivering mot principer

- **YAGNI (Beck 2000):** Variant B (freemium) inför Stripe/moms/VAT/ToS/DPA/näringsverksamhet utan validerad efterfrågan. Värdet är spekulativt; regulatorisk friktion är konkret.
- **Reversibility (Fowler 2002, "Last Responsible Moment"):** Variant C är enda alternativet som bevarar pivot-rätt i båda riktningar — C → A trivialt, C → B möjligt, C → anonym JobAd-katalog möjligt.
- **SRP / SoC (Martin 2017 kap. 7):** Kostnadsskydd och intäktsmodell är två change-reasons. Variant C skiljer dem rent.
- **Civic-utility-identitet (CLAUDE.md §1):** Variant B:s freemium-paywall bryter 1177/Digg-tonen. Inte estetik, identitet.
- **Defense in depth:** Auth-gated JobAd-yta är försiktigt default. Anonym-publik kan låsas upp senare som opt-in med IP-rate-limit + bot-skydd-mätning.

### Avvisade alternativ

**Alternativ A (stängd klassapp):** Domain-restriction (`@nbi.se`) är extra kod som inte återanvänds, reverse-pivot-kostnad icke-trivial, CV-narrativ försvagat. C uppnår samma kostnadskontroll utan låsningar.

**Alternativ B (public freemium):** Avvisad bestämt. YAGNI-brott + civic-utility-brott + regulatorisk friktion inkompatibel med studentstatus. Premature optimization på produktnivå.

**Anonym publik JobAd-katalog:** Avvisad i Fas 2-start. Auth-gated default. Anonym läs-yta kan låsas upp senare via separat ADR.

### Fas 4-koppling (TD-26)

Kostnadsskyddsmönstret från Fas 2 ärvs direkt:
- Ny `AiInvocationPolicy` (per UserId) läggs till på samma sätt som JobAd-policy idag
- `IAiOperation`-pipeline-behavior för token-tracking + per-user spend-cap (TD-26-design)
- Budget Actions-tröskeln höjs vid Fas 4 (per revision-historik 2026-05-09)
- Bedrock-IAM-auto-disable-mekaniken är redan på plats

Arkitekturell symmetri — Fas 2:s skyddsmönster är samma som Fas 4:s.

## Runda 2 — Amendment: invitations + waitlist

Klas-input efter Runda 1: registreringsflödet ska gå via (a) magic-link direkt till valda personer **och** (b) waitlist där Klas manuellt godkänner. CV-värde är **inte** beslutsaxel (Klas planerar private repo efter MVP).

### Beslut

**Accepterat scope-tillägg i Fas 2-prereqs med snitt mot Fas 6:**

| Del | Fas |
|---|---|
| Backend: Invitation-domän + Waitlist-domän + endpoints + email | **Fas 2 (nu)** |
| Frontend: `/vantelista`-publik signup-sida | **Fas 2 (nu)** |
| Admin-UI för waitlist + invitations | **Fas 6 (admin-panel-fasen)** |

Klas använder Postman/curl/Bruno mot admin-endpoints under Fas 2–5. YAGNI för 20 användare på 20 veckor (~1 invite/vecka).

### Domänmodell — två aggregates

```
Invitation (aggregate root)
├─ Id: InvitationId
├─ Email: Email (VO)
├─ Origin: InvitationOrigin { DirectInvite, WaitlistApproved }
├─ TokenHash: string (HMAC-SHA256, plaintext bara i email)
├─ ExpiresAt: DateTimeOffset (7 dagar default)
├─ Status: InvitationStatus { Pending, Redeemed, Expired, Revoked }
├─ IssuedByAdminId / IssuedAt / RedeemedAt / RedeemedByUserId
└─ Events: InvitationIssued, InvitationRedeemed, InvitationExpired, InvitationRevoked

WaitlistEntry (separat aggregate)
├─ Id: WaitlistEntryId
├─ Email: Email
├─ RequestedAt: DateTimeOffset
├─ Status: WaitlistStatus { Pending, Approved, Rejected }
├─ ApprovedAt / ApprovedByAdminId
├─ ResultingInvitationId: InvitationId? (sätts vid Approved)
└─ Events: WaitlistEntryRequested, WaitlistEntryApproved, WaitlistEntryRejected
```

Båda vägarna konvergerar på samma `redeem-invitation`-endpoint. Origin-fältet skiljer för audit-spår.

**Motivering:** Evans 2003 kap. 6 + Vernon 2013 kap. 10 — olika livscykler, olika invarianter, olika consistency boundaries kräver separata aggregates. `Approve()` på WaitlistEntry har idempotency-invariant ("Approved ⇒ exakt en Invitation") som hör hemma i aggregaten, inte i handler.

### Token-mekanism — dedikerad opaque token

- 32 bytes URL-safe base64 plaintext (skickas bara i email)
- HMAC-SHA256-hash lagras i DB
- Single-use via optimistic concurrency (RowVersion)
- 7 dagars expiry, konfigurerbar

**Avvisat:** JWT (deprecated per ADR 0017/0018, sessions har ersatt) + återanvänd session-token (annan livscykel, SRP-brott).

**Referens:** OWASP ASVS V3.7 — out-of-band token controls.

### Email-infrastruktur — SES via `IEmailSender`

`IEmailSender` i Application + `SesEmailSender` i Infrastructure. AWS SES (eu-north-1) redan dokumenterad subprocessor (BUILD.md §13.4) — ingen ny ADR behövs. Svenska email-templates per design-skills.

### `registrations_open`-flagga — behålls som kill-switch

`false` blockerar **både** invitation-redemption **och** waitlist-signup med 503 + Problem Details. Befintliga users opåverkade. Semantiken skiftar från "regulator för normal-flow" till "emergency brake".

### Rate-limit-policies — tre nya

| Endpoint | Policy | Limit | Nyckel |
|---|---|---|---|
| `POST /api/v1/auth/redeem-invitation` | `InvitationRedeemPolicy` | 5/h | per IP + per token (whichever first) |
| `POST /api/v1/waitlist` | `WaitlistSignupPolicy` | 3/24h | per IP |
| `POST /api/v1/admin/invitations` | `AuthWritePolicy` (befintlig) | 20/min | per IP |

**Avvisat:** Gemensam `RegistrationPolicy` 1/24h per IP — fel modell för invitations (single-use-token är redan stark spam-spärr).

### Admin-endpoints — Postman/curl/Bruno under Fas 2–5

- `POST /api/v1/admin/invitations` — utfärdar Invitation, skickar email
- `GET /api/v1/admin/invitations?status=Pending` — listar (debug)
- `POST /api/v1/admin/invitations/{id}/revoke`
- `GET /api/v1/admin/waitlist?status=Pending` — listar (Klas pollar manuellt)
- `POST /api/v1/admin/waitlist/{id}/approve` — skapar Invitation, skickar email
- `POST /api/v1/admin/waitlist/{id}/reject`

Skyddade av `[Authorize(Roles = "SuperAdmin")]` (BUILD.md §11.4).
