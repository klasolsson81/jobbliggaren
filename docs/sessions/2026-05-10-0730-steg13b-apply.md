## In-flight fixar (ej i original startprompt)

Tre kod-fixar krävdes mid-apply, dokumenterade och committade:

### Fix 1 — KMS-policy CloudWatch Logs grant (security)

**Problem:** 5 LogGroups failade med `AccessDeniedException: The specified KMS key does not exist or is not allowed to be used` vid första foundation-apply. Baseline-KMS-keyn (`alias/jobbpilot-master-key`) hade bara `EnableIAMUserPermissions` — saknade `Service`-principal-grant för `logs.eu-north-1.amazonaws.com`.

**Fix:** Patcha `infra/terraform/modules/kms/main.tf`:
- Lägg till `data "aws_region" "current" {}` 
- Lägg till `AllowCloudWatchLogsEncryption`-statement i `key_policy` som grantar `logs.${region}.amazonaws.com` Service-principal `kms:Encrypt*/Decrypt*/ReEncrypt*/GenerateDataKey*/Describe*` med snäv `kms:EncryptionContext:aws:logs:arn` ArnLike-condition

**Apply-konsekvens:** Krävde apply mot prod-stack (där baseline-KMS bor) innan dev kunde fortsätta. Auto-classifier blockade prod-apply som "scope escalation" — krävde Klas explicit GO. Apply: 0 add, 2 change, 0 destroy. In-place uppdatering av master + byok keys. Cost-impact $0.

### Fix 2 — ASCII-only Security Group descriptions

**Problem:** 4 SGs failade med `InvalidParameterValue: Value (...) for parameter GroupDescription is invalid. Character sets beyond ASCII are not supported.` AWS EC2 GroupDescription accepterar inte åäö.

**Fix:** Replace-all `"från"` → `"from"` i `infra/terraform/modules/network/main.tf` (11 förekomster i 4 SG + 7 ingress/egress-rule descriptions).

**Lärdom:** AWS API-fält som accepterar UTF-8 vs ASCII-only är inkonsistent dokumenterat. Default till ASCII för alla AWS-resource-`description`-fält som inte är UI-copy. Kommentarer i .tf-filer påverkas inte.

### Fix 3 — Dockerfile base-image (.NET 10 noble)

**Problem:** Docker build failade med `not found: mcr.microsoft.com/dotnet/sdk:10.0-bookworm-slim`. .NET 10 GA bytte default base-image från Debian 12 (Bookworm) till Ubuntu 24.04 (Noble).

**Fix:** Replace `10.0-bookworm-slim` → `10.0-noble` i bägge Dockerfiles (Api + Worker, sdk + aspnet stages).

**Verifierat via** `docker manifest inspect`: `dotnet/sdk:10.0-noble` + `dotnet/aspnet:10.0-noble` finns; `10.0-trixie-slim` (Debian 13) finns inte.

### Fix 4 — Secrets-version placeholders (post-foundation, post-fullstack)

**Problem:** ECS task-startup failade med `ResourceNotFoundException: Secrets Manager can't find the specified secret value for staging label: AWSCURRENT`. Secrets-resurser var skapade men utan version → ingen `AWSCURRENT`-label.

**Fix:** Lägg till 2 `aws_secretsmanager_secret_version`-resurser i `environments/dev/main.tf` med PLACEHOLDER-värden:
```
Host=${rds.address};Port=5432;Database=jobbpilot;Username=PLACEHOLDER_SET_BY_STEG_14;Password=PLACEHOLDER_SET_BY_STEG_14;SSL Mode=Require;Trust Server Certificate=true
```
+ `lifecycle { ignore_changes = [secret_string] }` så STEG 14 kan skriva över utan rollback.

**Konsekvens:** Application kraschar vid första DB-query (PLACEHOLDER-creds existerar inte i RDS) → Worker-task restartar i loop tills STEG 14 sätter riktiga creds. Förväntad transient state.

## Lärdomar STEG 13b apply

- **Pre-apply audit av AWS-API-konstrainter saknades.** ASCII-only `GroupDescription` + KMS-policy-Service-principal-grants är dokumenterade AWS-mönster som code-reviewer/security-auditor inte fångade i kod-fasen. Lärdom: vid första apply mot ny AWS-resurs-klass, läs senaste AWS-resource-spec före plan.
- **Cross-stack KMS-policy-dependencies.** Baseline-stacken (prod-environment) äger KMS, dev-environment konsumerar via alias-lookup. Konsumtion (`aws_cloudwatch_log_group` med `kms_key_id`) kräver att källan har grant — annars `AccessDeniedException` vid första task-skapning. Lärdom: KMS-policy-grants ska följa alla framtida konsumenter, inte bara nuvarande.
- **Microsoft-image-tag-konvention följer OS-LTS-cyklar.** .NET 10 default = Ubuntu 24.04 Noble (LTS), inte Debian 12 Bookworm. Verifiera image-existens via `docker manifest inspect` innan första build i ny .NET-major-version.
- **Secrets-resurs ≠ secret-version.** `aws_secretsmanager_secret` skapar containern, `aws_secretsmanager_secret_version` skapar AWSCURRENT-värdet. ECS task-startup kräver bägge. Vid placeholder-strategier: lifecycle.ignore_changes på secret_string så manuella `put-secret-value` inte rolls back av Terraform.
- **Force-new-deployment är manuellt steg vid deployment-circuit-breaker rollback.** ECS markerar service-deployment FAILED och återstartar inte automatiskt nya tasks utan operatör-trigger. Vid post-fix recovery: `aws ecs update-service --force-new-deployment`.
- **Auto-classifier scope-escalation-block är värdefullt.** Prod-apply blockerades trots att operationen var $0-cost in-place key-policy update. Friktionen tvingade explicit Klas-GO för prod-stack-mod, även när det är säkerhetshöjande.
