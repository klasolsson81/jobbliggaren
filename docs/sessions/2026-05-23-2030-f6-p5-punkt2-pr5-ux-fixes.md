### EF Core 10 batch-Contains-translation-bugg (commits `31cd5c5` → `3849976`)

**Symptom:** Application.UnitTests `UserStatus`-tester gröna lokalt, **CI rött** med Npgsql-translation-fel:

```
The LINQ expression '...Where(j => userJobAdIds.Contains(j.Id))' could not be translated.
```

**Rotorsak:** strongly-typed VO `JobAdId : record struct` har `IEquatable<JobAdId>` men EF Core 10 + Npgsql 10:s batch-genererare lyckas inte översätta `IReadOnlyCollection<JobAdId>.Contains(EntityProperty)` när entry-collection är liten (<=10 — tröskel under vilken Npgsql byter från PARAMETERLIST till constant-array-literal). Bug är specifik till EF Core 10-uppdateringen — Fas 1-koden (EF Core 9) använde samma mönster utan problem.

**Försök 1 (`31cd5c5`):** `.Select(x => x.Id).Where(...)` — Npgsql-translationen fortfarande röd, samma rotorsak.

**Försök 2 (`3849976`):** **client-side `HashSet<JobAdId>`-filter** efter projicerad SQL-fetch — EF läser bara JobAdId+overlay-shape, lookup sker i process. Acceptabel kostnad för publik batch-port (cap=50 IDs per batch, dokumenterat i ADR 0063 §Konsekvenser). **CI grön.**

**Memory uppdaterad:** `feedback_ef_strongly_typed_vo_contains_translation.md` — VO-`Contains` mot entity-property i EF Core 10/Npgsql kan kräva client-side eval; verifiera per query.

## TDs lyfta (§9.6-godkända)

### TD-87 (Major × F6 P5 P2-fas-stängning)

**Rate-limit för `/me/*`-batch-endpoints.** Nuvarande `MeReadPolicy` (per-user-FixedWindow) skyddar enskilda `/me/*`-endpoints men batch-porten (`/api/v1/me/saved-job-ads/batch` + `/api/v1/me/applications/by-job-ad/batch`) har **två amplifieringsvektorer** ej täckta av per-user-policyn:

1. Anonym-tolerant shape (ADR 0063 §Kontext-konsistens) → IP-axel-skydd krävs separat.
2. Batch-payload upp till cap=50 IDs → en request = upp till 50 lookup-operationer i process.

**Föreslagen åtgärd:** ny `MeStatusReadPolicy` med **dual-partition** (userId för autentiserade, IP för anonyma) + skarpare burst-cap (15/10s istället för MeReadPolicy 60/30s). Implementeras innan Punkt 2-fas-stängning räknas formellt klar.

§9.6-motivering: Major × F6 P5 P2-fas-stängning — denna fas, denna sprint, lyft enbart för att Klas-direktiv 2026-05-23 var "fortsätt non-stop till nästa Klas-stopp; TD acceptabelt för rate-limit-policy-nyans som inte hör till PR5 visual-verify-feedback-scope".

### TD-88 (Minor × Trigger)

**DOM-mutation `onMouseOver` i hero-chips → CSS `:hover`-refactor.** PR5 hero-chip-implementation använder React state + `onMouseOver`/`onMouseOut` för att toggla `data-state`-attribut för chip-bg-färg. CSS-only `:hover`-pseudoclass uppnår samma effekt utan render-cykel. Opportunistisk touch — utlöses nästa gång hero-chips eller liknande pattern refaktoreras.

## Lessons learned

1. **EF Core 10 + Npgsql 10 batch-Contains translation av strongly-typed VO mot entity-property kan kräva client-side eval.** Tröskel beror på collection-storlek (<=10 byter Npgsql till constant-array-literal). Memory: `feedback_ef_strongly_typed_vo_contains_translation.md`. Verifiera per query — Fas 1-koden var EF Core 9, så regressionen kom med EF Core 10-uppdateringen.
2. **Per-user-overlay på publik list-DTO är arkitekturprecedens, inte lokal fix.** ADR 0063 etablerar mönstret för framtida publik↔privat-domän-overlays (OG-cards, SSR-cache, framtida feeds). Att vidga `JobAdDto` direkt hade brutit cache-strategin ADR 0045 Beslut 1 anger.
3. **Klas visual-verify-feedback efter deploy är högsta värde-loop.** PR5+PR5b kostade ~3h totalt — 10 konkreta fynd som inte hade fångats av tester, design-reviewer-bilder eller code-reviewer. Inkludera i samtliga framtida punkt-stängningar.
